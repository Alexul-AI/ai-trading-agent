## Who the user is

Not a financial/trading expert — explain things in plain terms, not jargon. Wants the bot to eventually run with minimal ongoing effort from them (see goals below). Prefers direct, honest engineering assessments over cheerleading — has explicitly asked to be told when something won't work rather than encouraged to keep trying anyway.

## Goals

- Near-term: passive income above 200 ILS/month from this bot.
- Long-term: grow that with **minimal manual effort** — the bot should run unattended, not need babysitting.
- Planned capital path: $100 → $1,000 → $10,000, but **each increase should be earned by the previous tranche's track record, not by a calendar date**. See GOLIVE_CRITERIA.md.

## Hard boundaries (do not cross these)

- **Never execute a trade** (paper or live) on the user's behalf, even if explicitly asked — this is a standing rule, not a one-time refusal. Direct the user to do it themselves (Alpaca dashboard or the app's manual-order UI, currently disabled by the user on purpose).
- **Never give personalized investment advice** ("should I buy X", "is now a good time"). Give math, backtested facts, and mechanics; the decision is always the user's.
- Currently `TRADE_MODE=paper` and `AUTOPILOT_EXECUTE_TRADES=false` (dry-run only) — do not change either without the user explicitly asking, and treat flipping `AUTOPILOT_EXECUTE_TRADES` to `true` as a meaningful step worth confirming even though it's still paper money.

## Architecture facts that aren't obvious from the code alone

- `strategyEngine.ts`'s `decideTradeSignal` is the single source of truth for BUY/SELL/HOLD. Used live via `autopilotWorker.ts` and in `backtest.ts` / `backtest-sweep.ts` / `backtest-portfolio.ts`. Any strategy change must be reasoned about in all three backtest contexts plus live.
- **RSI/MACD warm-up bug**: both use recursive/Wilder smoothing seeded from `bars[0]` and need ~150 bars of runway before they're numerically accurate — a short series gives biased or literally-wrong values, not just "less precise" ones. This existed in three separate places and was fixed in all three: `backtest.ts`, `backtest-sweep.ts`, and the live ticker chart (`src/market/alpacaMarketData.ts` fetches extra warm-up history, `src/market/chartPoints.ts` trims after computing, not before). **If you add any new code that computes RSI/MACD over a bar series, apply the same pattern** — fetch/compute with 150+ bars of history before the window you actually care about, don't just fetch exactly what you plan to display. ATR (`indicators.ts` `calculateATR`) follows the same convention.
- Three BUY-side filters/toggles now exist, all **off by default on purpose**: `AUTOPILOT_SENTIMENT_FILTER` and `AUTOPILOT_INSIDER_FILTER` (unbacktestable - the underlying APIs only return current data, not point-in-time history - both fail open on fetch errors and cache 24h per ticker) and `AUTOPILOT_REGIME_FILTER` (backtestable, but results across two non-overlapping 900-day windows were mixed - see "Strategy performance reality" below). `useAtrStops` in `strategyEngine.ts`'s config is a related but separate toggle (also off by default) for ATR-based vs flat-percent stop-loss/take-profit distance.
- **Fractional/notional BUY fallback** (`AUTOPILOT_ALLOW_FRACTIONAL_SHARES`, added 2026-07-11, off by default): whole-share sizing (`Math.floor(cash / price)`) yields 0 shares for most/all of the 13 tickers below ~$1,000-3,000 of capital, silently blocking BUY signals at the user's actual planned starting capital ($100-$1,000). When enabled, a BUY that would otherwise size to 0 whole shares falls back to a notional dollar order instead (min $5, via `strategyEngine.ts`'s `suggestedNotional` field) - but that order does **not** get Alpaca's broker-side bracket stop_loss/take_profit (unconfirmed whether brackets work with fractional/notional sizing in Alpaca's API), relying solely on `decideTradeSignal`'s own cycle-based STOP_LOSS/TAKE_PROFIT re-evaluation instead. This is why it's opt-in like the sentiment/insider filters, not always-on like the bucket cap - it trades protection for capability, not a pure risk reduction. Whole-share sizing is completely unchanged whenever it yields >= 1 share, so this only ever affects behavior at low capital. Threaded through all four BUY-path layers: `strategyEngine.ts` (sizing) → `autopilotWorker.ts`'s `getSafeBuyNotionalForBucketCap` (bucket cap, dollar-based) → `server.ts`'s `executeSafeTrade` (builds a `notional` order payload with `time_in_force=day`, no `order_class=bracket`) → `riskManager.ts`'s `evaluateTrade` (dollar-based re-check, used by the shared manual+autopilot execution path).
- Portfolio-level risk layers, layered on top of the per-ticker 20%-equity cap: **circuit breaker** (`portfolioCircuitBreaker.ts`, always on, not a toggle) blocks new BUYs when equity drops `AUTOPILOT_MAX_DRAWDOWN_FROM_PEAK_PERCENT` (default -15%) from its peak - peak equity is *derived* each cycle from Alpaca's own `getPortfolioHistory` (not tracked in our own state), so a lost/corrupted local state file only costs a re-bootstrapped tracking window, not a wrong number. This state is **sticky**: once `tripped`, it stays tripped until a human calls `POST /api/autopilot/circuit-breaker/reset` - there is no auto-recovery when equity climbs back above the drawdown threshold. **Bucket concentration cap** (`getSafeBuySharesForBucketCap`, always on) caps combined exposure per asset-class bucket (`TICKER_TO_BUCKET`: `us_broad`/`international`/`bonds`/`commodities`/`high_beta_growth`) at `AUTOPILOT_MAX_BUCKET_EQUITY_FRACTION` (default 40%) - same bucket grouping is reused by the regime filter. Neither of these is a `useAtrStops`/`AUTOPILOT_REGIME_FILTER`-style opt-in; both are always-on downside-only guardrails, same category as the pre-existing per-ticker cap. **2026-07-13**: this bucket-cap/sell-throttle logic was extracted out of `autopilotWorker.ts` into `src/strategy/portfolioSafety.ts` (pure move, no behavior change - `autopilotWorker.ts` re-exports the same names so nothing else needed to change) specifically so it could be imported by a standalone script without dragging in `autopilotWorker.ts`'s transitive import of `agent.ts`, which constructs an OpenAI client at module load and throws if `OPENAI_API_KEY` is unset.
- A separate, **daily** (not peak-based) circuit breaker also exists: `riskManager.ts`'s `evaluateTrade` blocks all new BUYs when the account's equity is down more than `maxDailyDrawdownPercent` (hardcoded -5%, a different constant from the -15% peak breaker above) versus the previous day's close - checked at actual order-submission time in `server.ts`'s `executeSafeTrade`, not during signal generation. Binds far more often than the -15% peak breaker in practice, and is easy to forget about since it lives in a different file with no env var to grep for.
- **Portfolio-level backtest** (`backtest-portfolio.ts`, added 2026-07-13): unlike `backtest.ts`/`backtest-sweep.ts`, which simulate each ticker independently with its own `$10,000` capital, this one runs all 13 tickers through a single shared cash pool with the bucket cap, sticky circuit breaker, sell-fraction throttle, and daily -5% kill switch all actually applied - the only tool in the repo that can answer "does the strategy still look reasonable once it's competing for one real account's capital." Reuses `getSafeSellShares`/`getSafeBuySharesForBucketCap` from `portfolioSafety.ts` and `evaluateTrade` from `riskManager.ts` directly rather than re-deriving the caps. Models live's actual two-phase behavior: every ticker's sizing decision that day reads one frozen start-of-day snapshot (matching `autopilotWorker.ts`'s real per-cycle concurrency, where all tickers read the same pre-cycle snapshot), then SELLs execute before BUYs (a ticker later in the buy queue can be limited by cash an earlier same-day sell just freed, matching live's per-order real-cash re-check). Runs a 4-variant ablation each time (A: no caps at all / B: bucket cap only / C: +circuit breaker / D: +daily kill +sell throttle = full system) to isolate what each layer actually costs/protects, and writes a full report (`data/backtest-reports/` - gitignored, local only) with a markdown summary plus `blocked-signals.csv`/`trades.csv`/`equity-curve.csv` for the full-system variant, including forward 5d/20d returns on every blocked BUY signal. The circuit breaker's stickiness (never auto-recovers without a manual reset) is shared, tested logic (`applyStickyTrip` in `portfolioCircuitBreaker.ts`, used by both live and this backtest) rather than re-implemented separately. Fractional/notional sizing isn't supported (whole shares only). **2026-07-13, second pass**: originally shipped close[t]-execution only (a same-bar assumption shared with `backtest.ts`/`backtest-sweep.ts`); now also runs a `NEXT_OPEN` companion model in the same invocation (signal on close[d], executed at open[d+1] instead) specifically to check whether that assumption was inflating the numbers - see the next bullet for what it found. `backtest.ts`/`backtest-sweep.ts` still close[t]-only, unchanged - next-open stayed scoped to this one script on purpose. **Because every variant shares that same close[t]-execution assumption, none of the return numbers below are a realistic live-return forecast - they're only valid for comparing variants against each other.** Two metrics are easy to conflate and mean different things: `bucketCapReductionCount` (times the cap reduced a BUY request, partially or fully) vs. `blockedBuyCount` (BUY signals that executed literally 0 shares, for any reason) - the first counts partial shrinks too, the second doesn't. First real run (900-day window, 406 simulated days): full-system return +20.66% (max DD -11.43%, bucket cap reducing a request 184 times - of which 158 were cut all the way to 0 shares - circuit breaker never tripping) vs. +18.31% with the daily-kill-switch/sell-throttle layer removed (variant C) vs. +29.83% with every cap removed (variant A) vs. independent-per-ticker-average +3.00% vs. equal-weight buy-and-hold +41.68%. Notably **not monotonic**: variant D (full system, most restrictive on paper) had *more* trades (673) and a *higher* return than B/C (333 trades). Precise framing, not "protection improved returns": the sell-fraction throttle changed exit dynamics in this window (forcing partial exits instead of one full exit), which happened to let winners keep running longer - an observed result in this specific window, not a validated edge. One run on one window - informative, not yet a validated finding (same standard as everything else in "Strategy performance reality" below).
- **Next-open execution model result (2026-07-13)**: on the same 900-day window, delaying execution from close[d] to open[d+1] (see previous bullet) is a real, meaningful drag even on the raw strategy - variant A (no caps) fell from +29.83% to +19.54% (max DD -11.67% -> -14.80%), and B/C (bucket cap, +circuit breaker) fell from +18.31% to +7.83%. But full-system (D) didn't just drag, it inverted: +20.66% -> **-9.68%** (max DD -11.43% -> -16.17%). The mechanism isn't "the daily -5% kill switch fires more" (it fired **zero** times in either model, confirmed in the report) - it's that next-open's choppier equity path tripped the **-15% peak circuit breaker** on 2025-04-08, something that **never happened at all** under close[t]-execution. Once tripped, the sticky no-auto-recovery design (see two bullets up) locked out new BUYs for 315 of 406 days (78%) - SELLs kept firing (never blocked by the breaker), so the portfolio just drained toward cash for the rest of the window without re-entering. 561 of the 628 blocked BUY signals were circuit-breaker blocks specifically (only 67 bucket-cap), and `blocked-signals.csv`'s forward-return column shows some of those blocked signals would have been profitable (e.g. a blocked TSLA BUY on the trip date returned +14.5%/+24.5% over the next 5/20 days it never got to capture). **Read carefully**: this is not "the strategy is proven broken" - it's "an execution-timing assumption and an always-on safety layer interact in a way that was invisible under the old backtest model, because the breaker never got the chance to trip there." It raises a real, so-far-unanswered question about whether -15%-peak-with-no-auto-recovery is well-calibrated, or whether it needs a monitoring/alerting nudge so a human notices and considers a reset rather than staying locked out silently for most of a year. One window - the next step (multi-window validation, already on the backlog) matters more now than before this ran, not less.
- **Circuit-breaker intervention scenarios (2026-07-13)**: direct follow-up to the finding above - simulates (backtest-only, nothing live changed) what would have happened if a human had reset the breaker instead of it staying sticky forever. Reuses live's actual reset semantics exactly: `resetPortfolioCircuitBreaker(currentEquity)` (`portfolioCircuitBreaker.ts`) rebases peak equity to the equity *at the moment of reset*, not the original pre-trip peak - a naive "just clear the tripped flag" would have re-tripped almost immediately and produced a meaningless result. Same 900-day next-open window, variant D only (C never trips in any run, so the axis is meaningless there): the current sticky/no-reset policy (D0) gave **-9.68%**. Every single alternative tested turned it positive - reset 1 trading day after trip: **+8.14%**; 3 days: +6.55%; 5 days: +6.62%; 10 days: +8.12%; reset once drawdown recovers above -10% from peak: +5.02% - **and max drawdown was identical (-16.17%) across every one of them**, including D0. That last point matters most: in this window, waiting longer to reset bought no risk reduction at all versus resetting fast, it only cost forfeited upside (each policy's blocked BUYs averaged +4-12% over the following 20 days). Every reset-based policy re-tripped at most once (`resetCount` was 0 or 1 across the board) - this wasn't a policy that has to fight the breaker repeatedly. Read precisely: this is one window, and "no reset policy made drawdown worse here" is not the same as "no reset policy ever could" - but the consistency across five very different reset rules (1/3/5/10-day and recovery-based) landing in the same +5-8% range, with identical downside, is a stronger signal than most single-window findings in this file get. Multi-window validation is the natural next check before touching anything live.
- **Multi-window validation result (2026-07-13) - the single-window finding above does NOT generalize, and reverses in a real bear market.** `backtest-portfolio-multiwindow.ts` (new) reuses the exact same simulation via an extracted `runWindowAnalysis` function and ran the same ablation + circuit-breaker-policy comparison across 5 windows: current (~900d), the prior ~900d, 2022 bear-heavy, 2023-2024 bull-heavy, and COVID crash+recovery (2020-2021). The circuit breaker only actually tripped under D0 in **2 of the 5 windows** - the other 3 never tripped at all under next-open, so there was nothing to compare in them (not evidence for or against reset policies, just no signal). Of the 2 windows where it did trip: **current window confirms the earlier finding** (every reset policy beat D0's -9.68%, no worse drawdown). **2022 bear-heavy inverts it**: D0 gave -12.38% (287 halt days); the four time-based reset policies (1/3/5/10 trading days) all had a *slightly better return* (-9.79% to -11.43%) but a **meaningfully worse max drawdown** (-21.96% to -23.69%, vs D0's -15.89%) - resetting early let the strategy re-enter positions that then got hit again by the market continuing to fall, digging the drawdown deeper than staying in cash would have. The recovery-based policy (D3, reset only once drawdown improves above -10%) never triggered in this window (0 resets) and matched D0 exactly - in a real sustained bear market, drawdown never recovered enough to earn a reset, which is arguably that policy working as intended, not failing. **Bottom line so far**: reset-by-fixed-timer looks good after a moderate/mixed drawdown but can make a genuine bear market's drawdown meaningfully worse; recovery-gated reset didn't show that failure mode in either tripped window, which makes it the more promising direction of the two, but this is still only 2 data points where the breaker actually fired - nowhere near enough to change anything live. Full numbers in `data/backtest-reports/multi-window-summary.csv` (gitignored, local only) after running the script.

What was added: (1) a once-per-calendar-day Telegram reminder while halted (`lastReminderSentDate` field + pure `shouldSendDailyReminder` helper in `portfolioCircuitBreaker.ts`, checked once per cycle using that cycle's own decisions to report today's blocked-BUY count - no extra journal read needed for that part), (2) a new append-only audit log (`circuitBreakerAuditLog.ts` -> `data/circuit-breaker-audit.jsonl`, three event types: `CIRCUIT_BREAKER_TRIPPED`/`REMINDER_SENT`/`RESET`), (3) a required `reason` field on the reset endpoint (400 on missing/blank - a manual-only reset means there is always a reason on record) plus a Telegram reset confirmation, and (4) a new public `GET /api/autopilot/circuit-breaker/review` endpoint (haltedAt, peak/current equity, drawdown, threshold, daysHalted, blockedBuyCountSinceHalt, last 10 blocked signals, positions, cash) - all derived from data that already existed, nothing new tracked. - **Minimal research scorecard** (`scorecard.ts`, added 2026-07-14, `docs/product/ROADMAP.md` Phase 1): a single standard for comparing any strategy backtest against a benchmark - CAGR, max drawdown, Calmar ratio, exposure, trade count, benchmark comparison (SPY / equal-weight buy-and-hold). Deliberately minimal per the roadmap - Sharpe/Sortino/profit factor/expectancy are explicitly deferred, not part of this module. **CAGR and Calmar did not exist anywhere in this repo before this** - benchmarks were previously compared as raw total-return-over-the-window percentages, not annualized, so a 900-day window and a 305-day window weren't actually comparable. Pure, generic functions (`computeCagrPercent`, `computeCalmarRatio` - take primitives, not `PortfolioSimResult`) so this is reusable by the future ETF Rotation backtest (Phase 2) without depending on `backtest-portfolio.ts`'s specific result shape - same pure/adapter split as `applyStickyTrip` in `portfolioCircuitBreaker.ts`. Wired into both `backtest-portfolio.ts` (a new "## Scorecard" section in the existing per-execution-model markdown report) and `backtest-portfolio-multiwindow.ts` (a new "Scorecard by window" table, NEXT_OPEN variant D only - the roadmap's "next-open as primary result" - appended as a fourth block in `multi-window-summary.csv`). A real run surfaced a genuine interpretability trap worth knowing about: annualizing a short window's return produces a mathematically-correct but misleading CAGR (a 41-day +4.08% return annualized to +42.71% in one real single-window run) - `formatScorecardMarkdown` now appends an explicit caveat whenever the window is short. **2026-07-14, correction (caught in code review before merge, not self-caught)**: the first version annualized using `simDays` = `commonDates.length - simStartIndex`, which is a count of Alpaca daily-bar trading sessions (no weekends/holidays), not calendar days - so `365/simDays` systematically overstated CAGR magnitude (a trading-day count understates true elapsed time by roughly 365/252, ~1.45x). Fixed by adding `calendarDaysInclusive(startDate, endDate)` and threading a separate `annualizationDays` (calendar days) alongside `simTradingDays` (the original trading-bar count, kept for display/context) through `ScorecardMetrics`/`BenchmarkMetrics` - CAGR now always annualizes over calendar time. Corrected real numbers (same 5-window run): "Current (~900d)"'s -9.68% total return over 406 trading days spans 596 calendar days (2024-11-25 to 2026-07-13) and annualizes to **-6.04% CAGR** (not the originally-reported -8.75%) - confirmed by hand, `(1-0.0968)^(365/596) - 1 ≈ -6.05%`. Calmar for that window moved from -0.54 to -0.37 accordingly. The short-window caveat threshold and all report prose now check `annualizationDays`, not trading-bar count.
- **ETF Rotation MVP** (`etfRotationStrategy.ts` + `backtest-etf-rotation.ts`, added 2026-07-14, `docs/product/ROADMAP.md` Phase 2): the first real income-strategy candidate, chosen to go first ahead of Trend Participation for lower turnover/concentration risk. Pure decision module (`decideRotationTargets`, `computeMomentumReturnPercent`, `passesTrendFilter`, `isMonthlyRebalanceDate` - same pure/no-I/O convention as `decideTradeSignal`/`applyStickyTrip`) ranks a 5-ETF universe (`SPY`/`QQQ`/`EFA`/`TLT`/`GLD` - US broad/growth-Nasdaq/international/bonds/gold) by 126-trading-day trailing momentum, keeps the top 2 that also pass a SMA(200) trend filter, and equal-weights them at a **fixed** 100/holdCount% per slot - a pick failing its trend filter is dropped and that slot's weight goes to cash, never redistributed to the surviving pick (a real bug caught by the module's own tests before it shipped: the first implementation redistributed a failed slot's weight to survivors, which is backwards from "get more defensive in a downtrend"). Rebalances monthly (first trading day of each calendar month, not a fixed trading-day counter - avoids drift). Reuses `calculateSMA` from `indicators.ts`. This backtest does not model bucket cap/circuit breaker/daily kill switch, because the first goal is to measure the clean rotation strategy on diversified ETFs - **this does not mean these platform-level protections are disabled or considered unnecessary for future paper/live execution**; even an ETF portfolio can hit a prolonged bear market, and the portfolio circuit breaker should stay a platform-level safety net regardless of which strategy runs on top of it. Each rebalance fully liquidates and rebuys rather than trading only the delta (documented MVP simplification, inflates trade count somewhat). Runs both `close_to_close` and `next_open` execution models per Phase 1's now-standing rule, and uses `scorecard.ts` for CAGR/Calmar/benchmark reporting. **Uses raw Alpaca bars (`adjustment=raw`)** - ETF dividends/distributions (SPY/EFA/TLT all pay meaningful yields; GLD doesn't) aren't included, so both the strategy's and the benchmarks' returns are price-return, not total-return. The strategy-vs-benchmark comparison is still apples-to-apples (both miss dividends equally), but the absolute numbers below aren't real total-return figures - a future research pass should switch to adjusted/total-return data before treating this as a real income candidate. **First real run** (same "Current ~900d" window used throughout this project's research, 2024-11-25 to 2026-07-13, 406 trading/596 calendar days): next-open total return +26.11% (CAGR +15.26%, max DD -13.43%, Calmar 1.14, 21 rebalances, 80 trades) vs. close-to-close +26.92% (CAGR +15.72%) - only a ~0.8pt difference between execution models, much smaller than the daily-signal baseline strategy's ~30pt next-open drag on this same window, because monthly rebalancing is inherently far less sensitive to a one-day execution lag than a daily-signal strategy is. Both beat the window's own benchmarks (SPY CAGR +14.86%, equal-weight-5-ETF CAGR +14.74%) *and* dramatically beat the existing baseline strategy's next-open CAGR on this identical window (-6.04%, see the scorecard-fix bullet above). **Read carefully**: this is one window, one run - not yet multi-window validated, which is this project's own explicit standard (Phase 1) before treating anything as a real finding, not a foregone conclusion. A multi-window companion script is the natural next step, same phased approach as `backtest-portfolio.ts` (single-window first, multi-window later after real numbers exist to justify the extra validation work).
- **ETF Rotation multi-window validation result (2026-07-14) - beats the existing baseline strategy on return in most windows, but the 2022 bear-heavy window flags a real concentration-risk concern.** `backtest-etf-rotation-multiwindow.ts` (new, mirrors `backtest-portfolio-multiwindow.ts`) ran the identical 5 windows used for the baseline strategy's own multi-window study, so the two are directly comparable window-for-window (next-open, both raw-price - see the dividend caveat above). Against the **existing baseline strategy's** next-open return on the same windows (not just SPY): Current +26.11% vs -9.68%; Prior +11.74% vs +8.66%; 2022 bear-heavy -12.72% vs -12.38%; 2023-2024 bull-heavy +23.55% vs +12.70%; COVID +4.90% vs +1.82%. **ETF Rotation beats the baseline strategy's return in 4 of 5 windows** (often substantially), and is roughly a wash in the 5th (2022 bear-heavy, -12.72% vs -12.38%, ETF Rotation marginally worse on return). Max drawdown tells a more mixed story: ETF Rotation's drawdown is *smaller* than baseline's in 3 of 5 windows (Current -13.43% vs -16.17%; Prior -9.97% vs -14.28%; COVID -6.24% vs -7.04%) but **meaningfully larger in 2022 bear-heavy** (-22.08% vs -15.89%, about 6pts worse) and slightly larger in 2023-2024 bull-heavy (-10.31% vs -8.16%). Against SPY specifically (not the baseline strategy), ETF Rotation only wins in 1 of 5 windows (Current) - it trails SPY by a wide margin in both strong-bull windows (2023-2024 bull-heavy: +23.55% vs SPY's raw-price +35.78%; COVID: +4.90% vs SPY's +13.25%) because holding only 2 concentrated ETFs with a defensive trend-filter fallback inherently gives up upside during a market-wide rally that a 100%-invested SPY position fully captures. | Layer | Default | Scope |
|---|---|---|

| Bucket concentration cap (40%) | always on | caps combined exposure per asset-class bucket |

## Strategy performance reality (from backtesting done 2026-07-09, extended 2026-07-11)

- Baseline strategy: roughly +1.4% to +2%/year average across 10 large-cap tickers, highly ticker-dependent (-1.4% to +6%).
- Every parameter tweak tried on 2026-07-09 either did nothing real (turned out to be a warm-up-bug artifact) or was a net loss on a risk-adjusted basis (wider position sizing is just leverage, not edge; a cooldown after stop-loss made things worse; trend filters only "worked" by overfitting to specific tickers).
- Buy-and-hold massively outperformed the strategy on trending tickers in the same window (e.g. AMD +206% vs. the strategy's single-digit return). This strategy trades upside for smaller drawdowns by design — it is not a growth strategy, and that tradeoff should inform any discussion of expected returns.
- **2026-07-11**: ticker universe expanded from 5 correlated tech names to 13 spanning sectors/asset classes (`AMD,NVDA,AAPL,MSFT,TSLA,JPM,JNJ,XOM,PG,SPY,GLD,TLT,EFA`) - validated on the same 900-day window, average return held in the same ~1-3%/yr ballpark while the new sector-spread tickers showed meaningfully smaller max drawdowns than the tech-heavy five.
- **ATR-based stops** (`useAtrStops`): first calibration (2.5x stop / 4.5x take-profit ATR multiplier) was a clean net negative (avg return +1.64% vs +3.05% baseline). A wider 3.5x/6x calibration roughly matched baseline on return/drawdown while meaningfully improving win-rate consistency on the most volatile tickers - "roughly matches baseline" wasn't judged strong enough evidence to flip the default on, so it ships off with the 3.5x/6x numbers as `DEFAULT_STRATEGY_CONFIG`'s (unused-unless-enabled) defaults.
- **Portfolio regime filter** (`AUTOPILOT_REGIME_FILTER`): tested on two non-overlapping 900-day windows (one of which - confirmed via SPY's own buy-and-hold, -25.4% max drawdown - included the 2022 bear market) with three variants (exempt high-beta tickers from the filter / don't exempt them / gate high-beta tickers by the broad market's own regime instead of a self-referential composite). Drawdown improved consistently across all three variants and both windows; the effect on *return* was inconsistent between windows and between variants - no clean winner, ships off.
- Don't re-propose "just widen the position size," "just add a trend filter," "just add ATR-based stops," or "just add a regime filter" as improvements without checking `GOLIVE_CRITERIA.md` and this history first — all were already tried and either rejected or shipped off-by-default with mixed/inconclusive evidence.
- **2026-07-13**: `backtest-portfolio.ts` (see "Architecture facts" above) ran the shipped default config as one real portfolio for the first time, instead of 13 independent per-ticker runs. ## Deliberately not built (and why, so it isn't re-litigated from scratch)

- **Fundamentals-based BUY filter**: timescale mismatch. P/E and dividend yield change quarterly; this strategy holds positions for days to weeks. Either the filter never fires, or it permanently blocks the exact high-growth tickers (high P/E) that back-tested best.
- **Backtesting the sentiment/insider/fundamentals signals**: not possible with current data sources — Alpaca News, Alpha Vantage, and SEC EDGAR only expose current state, not a point-in-time historical snapshot.
- **Risk-based position sizing** (size by $ risk to stop-loss distance, instead of a flat % of equity): deferred until capital scales up, not rejected. Real prices checked 2026-07-11 against the actual 13-ticker universe: at $100 capital, the 20% single-position cap ($20) can't buy even 1 whole share of any of the 13 tickers (cheapest is TLT at ~$85) — the bot cannot open a position at all at this tranche, regardless of sizing formula. At $1,000 (cap $200), only 4 of 13 (TLT/EFA/XOM/PG) are buyable, and only as a binary 1-share-or-0 choice — there's no gradient for a smarter formula to size. Only at the $10,000 tranche does share count reach a real range (roughly 2-23 shares across the universe at then-current prices) where risk-based sizing would actually produce a different result than the flat equity-fraction cap. Revisit when capital reaches that tranche, not before. (The "no fractional shares anywhere in the codebase" blocker this note originally referenced was addressed the same day - see `AUTOPILOT_ALLOW_FRACTIONAL_SHARES` above - but risk-based sizing itself is still not worth building until share counts are large enough for a formula to matter.)
